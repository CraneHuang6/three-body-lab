# 模拟器无限向前物理轨迹设计

## 目标

把模拟器当前“预计算一段缓存后循环播放”的行为，改成“按需继续向前扩展物理轨迹”，使默认模拟和模拟器预设都能在当前初始条件下无限向前运行。

## 非目标

- 不改观演模式的预计算播放模型。
- 不改现有自动重置规则，仍然只有 `aliveCount === 0` 时自动重置。
- 不把模拟器改成完全实时逐帧积分。

## 当前问题

- `createSimulationCache()` 只在初始化时按 `duration * simSpeed + 0.5` 预计算固定长度的 `snapshots`。
- `advanceSimulatorTime()` 在到达 `duration` 后回绕到 0，因此默认模拟只能重复播放已有缓存，而不能继续向前产生新轨迹。

## 方案

采用“增量扩展缓存”：

1. 保留现有缓存采样与尾迹渲染模型。
2. 新增一个“从最后一帧继续积分”的扩展入口。
3. 当播放时间接近现有缓存尾部时，自动向后追加一段新的 `snapshots`。
4. 新追加的积分必须继承当前最后一帧的真实存活状态、位置、速度、质量，以及已有碰撞结果。

## 实现边界

### 1. 仿真层

- 在 `src/lib/simulation.jsx` 中新增一个基于现有 `simulate()` 的扩展函数，用于从尾帧继续向前积分固定步数。
- 追加时不能把已有 `collisions` 覆盖掉；新碰撞应以时间偏移后的绝对 `t`/`step` 追加到原数组末尾。
- `sampleSimulationCache()` 与 `getTrailFromCache()` 保持读取接口不变。

### 2. 模拟器层

- 在 `src/modes/SimulatorMode.jsx` 中维护一个可增长的 cache state，而不是只依赖一次性的 `useMemo(createSimulationCache(...))`。
- 当 `simTime` 接近当前缓存末尾时触发一次扩展，阈值应留出安全余量，避免播放时间先越界再补算。
- 扩展只应发生在模拟器模式；任何参数变化仍然重建全新缓存并把时间归零。

### 3. 验证

- 为仿真层新增测试，证明扩展后的 cache 长度增长，且扩展前后的尾帧连续。
- 为播放层新增测试，证明超过原始 `duration` 后不会回绕到开头，而是继续向前采样扩展后的轨迹。

## 风险

- 如果扩展逻辑重复从旧尾帧积分，可能出现碰撞重复记账。
- 如果扩展触发太晚，UI 可能短暂采样到缓存末帧并表现成停顿。
- 如果把“扩展时间”和“展示时间”耦合错，尾迹可能断裂。

## 推荐实现顺序

1. 先给 `simulation.jsx` 增加 cache 扩展能力与测试。
2. 再把模拟器缓存从 `useMemo` 切成可增长 state。
3. 最后调整播放时间逻辑，使其不再按 `duration` 回绕，而是依赖 cache 实际可采样范围继续前进。
